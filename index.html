The Solution

Decoupled Upload/Inference via Eventarc + Vertex AI Endpoint

Upload Now, Classify in the Background

Upload Service (Cloud Run, always-on) validates, stores to GCS, writes PENDING to Firestore, and returns requestId immediately — <500ms. GCS finalize event triggers Inference Worker via Eventarc. Worker preprocesses image (224x224, MobileNetV2 scaling), calls a warm Vertex AI Prediction endpoint (MobileNetV2 persistent, no cold starts), decodes top-3 ImageNet labels, updates Firestore COMPLETED. React polls /status every 2s until terminal state.

The upload and the classification are completely separate steps. The user gets an instant response with a tracking ID, then the AI processes the image in the background. A Vertex AI endpoint keeps the model warm at all times so there’s never a cold start delay.

The Problem

Multi-vector Prompt Injection in Production RAG APIs

AI APIs Vulnerable to Hidden Instructions

LLM-powered APIs face injection attacks via multiple surfaces simultaneously: user prompts override system instructions, uploaded PDFs carry malicious directives as RAG context, earlier conversation turns poison later ones (multi-turn/delayed attacks), and the model may leak system prompts or API keys in its output. Existing solutions scan only the latest message — missing distributed attacks across turns.

AI systems can be tricked in ways that aren’t obvious: attackers hide instructions inside uploaded PDFs, plant a command in an early chat message that activates later, or ask the AI to reveal its secret configuration. Standard defenses only look at the current message and miss these patterns.

The Solution

6-Layer Modular Guard System with Full Conversation Analysis

Six Independent Security Checkpoints

Pattern-based guardrail package (books_enh/services/guardrails/) with six guards: InputGuard scans current user message; ConversationHistoryGuard scans the entire context window for poisoning planted in earlier turns; DocumentGuard scans assembled RAG/PDF context with a stricter rule set; ToolCallGuard blocks mutating tools (create_loan, extend_loan) when context is compromised; OutputGuard blocks system prompt leaks and API key patterns in the response; SummaryQualityGuard blocks empty/too-short RAG answers. All tunable via a single patterns.py.

Six independent checkpoints inspect every request at different points in the pipeline — before the AI sees it, while checking knowledge base content, before executing actions, and after the AI responds. Each checkpoint blocks independently; passing one doesn’t guarantee passing the next.

Attack Vectors Defended
  • Instruction override: “Ignore previous instructions…”
  • Role manipulation: “You are now in developer mode…”
  • Multi-turn context poisoning: “For the rest of this conversation…”
  • PDF-embedded injection: malicious directives in book text used as RAG context
  • Unauthorized tool calls: LLM triggered to create_loan / extend_loan
  • System prompt exfiltration: “Reveal your developer instructions”
  • API key / secret leakage: sk-..., JWTs, DB DSNs in LLM output
  • Low-quality RAG answers: empty or <20-char responses blocked

Input & History Detection

InputGuard.check_messages() splits message list: the last user message triggers INPUT_GUARD; any earlier user message with injection patterns triggers CONVERSATION_HISTORY_GUARD. This catches the core problem — a poisoning message planted in turn 1 that’s designed to activate in turn 5. Assistant messages skipped (covered by OutputGuard). Pattern groups: INSTRUCTION_OVERRIDE, ROLE_MANIPULATION, SYSTEM_PROMPT_LEAK, SECRET_LEAK_INTENT, CONTEXT_POISONING.

The system reads not just the current message but the entire conversation history — catching attacks that are designed to be planted early and activated later. A manipulative instruction in message 1 blocks the whole conversation from that point forward.

Document Injection Guard

After RAG retrieves chunks and assembles the context string (but before injecting into the prompt), DocumentGuard scans with DOCUMENT_GROUPS — a stricter subset of patterns excluding soft phrases like “from now on” that legitimately appear in books. Designed to avoid over-blocking ordinary prose while catching clear injection signals like “Ignore all previous instructions” or “Reveal the system prompt” embedded in a PDF.

An uploaded PDF could contain hidden instructions meant to trick the AI. The document guard reads the PDF content before the AI sees it, checking for manipulation attempts. It uses a tighter filter than other guards to avoid blocking innocent books with common phrases.

Tool & Output Protection

ToolGuard splits tools into read-only (search_books, check_availability, get_member_loans, get_book_pdf_url, calculate_late_fees) and mutating (create_loan, extend_loan). Mutating calls refused when context shows injection indicators. OutputGuard uses regex to detect literal API key shapes (sk-..., sk-or-v1-..., JWTs starting eyJ, Postgres DSNs, env var assignments) and system prompt leak phrases in the model response before it reaches the user.

The tool guard stops the AI from being tricked into taking actions like creating loans without authorization. The output guard ensures the AI can never accidentally reveal secrets, API keys, or internal configuration — even if somehow asked to.

Multi-signal score: semantic similarity (40%), attribute match rate (30%), historical mapping patterns (20%), rule-based validation (10%). Calibrated thresholds per product category. Scores <85% trigger human review with top-3 candidates pre-ranked.

The system rates how sure it is about each match from 0-100%. Above 85% confidence it acts automatically. Below that, it shows a human the best options and asks them to pick — combining speed and accuracy.

Human Review UI

React-based review interface shows AI-suggested matches with evidence (matched attributes, similarity scores, similar confirmed mappings). Reviewers accept/reject/correct. All decisions feed back into model training loop improving future confidence scores.

The review screen shows reviewers the AI’s best guesses with explanations, making it fast to confirm or correct. Every human decision teaches the AI to do better next time.

Architecture Decisions

The core insight is decoupling upload from inference via Eventarc: the upload path is a lightweight HTTP service (512Mi Cloud Run) while inference runs in a separate 1Gi function triggered by the GCS event. This means upload latency is always <500ms regardless of model inference time. Vertex AI’s persistent endpoint eliminates the 5–15s cold start that in-function TensorFlow causes. Defense-in-depth auth (API Gateway validates key, Upload Service middleware re-validates) means neither service is exposed unprotected.

The main design win is that uploading and classifying are completely separate, so users never wait for the AI. The model runs on dedicated infrastructure that stays warm between requests. Infrastructure is fully reproducible — one Terraform command recreates everything.

Design Philosophy

The single most important decision: all detection patterns live in one file (patterns.py) using grouped sets (INPUT_GROUPS, DOCUMENT_GROUPS, OUTPUT_GROUPS). Adding a new attack pattern means editing one list in one file — no other file changes. The guards use direct-child class selectors from this file, so the update propagates automatically. GUARDRAILS_DEBUG=true adds a safe {stage, matched_category} diagnostic to blocked responses — never full prompts, PDF text, or secrets. Input-side blocks (5 of 8 attack vectors) fire before any LLM call, meaning blocked requests cost zero API credits.

All attack detection rules live in one file — so adding new protection is one line in one place, and every guard picks it up automatically. Most attacks are stopped before they ever reach the AI, meaning there’s no API cost for blocked attacks. The debug mode shows exactly which rule was triggered without revealing any sensitive content.
